# 用户流失与沉默定义及 SQL 查询文档

> 生成日期：2026/06/30
> 对应项目：女3_ProGoddessIII（game_id = 312）
> 数据来源：`gamelog_odl`（T+1 稳定）、`gameeco_raw`（角色快照）

---

## 1. 定义约定

| 用户状态 | 定义 | 判断依据 |
|---|---|---|
| **活跃用户** | 观察窗口期 N 天内有登录记录 | `MAX(login_ds) >= 起始日期` |
| **流失用户** | 观察窗口期 N 天内**没有任何登录**记录 | `MAX(login_ds) < 起始日期` 或 `COUNT(login) = 0` |
| **沉默用户** | 历史曾付费，观察窗口期 N 天内**有登录但无充值** | 快照 `total_pay_money > 0` 且窗口期 `login > 0` 且 `pay = 0` |
| **付费用户** | 截至观察期末，快照 `total_pay_money > 0` | `total_pay_money > 0` |
| **免费用户** | 截至观察期末，快照 `total_pay_money = 0` | `total_pay_money = 0` 或 无充值记录 |

**关键规则**：
- 流失的判断只看**登录流水**，不看充值流水。
- 付费用户只要还在登录，即使 N 天内没有充值，也归类为**沉默**，不属于流失。
- 只有“N 天内无登录”才判定为流失。

> 注：本版 SQL 使用 `snap_rolecache.total_pay_money` 作为历史累计充值判断依据。该字段与 `payrecharge` 全历史聚合在部分角色上存在差异（可能由 role_id 类型转换、脏数据或统计口径导致），在运营召回场景下使用快照字段更高效且稳定。如需精确到分，可改用 `payrecharge` 全历史聚合，但需拆步执行以避免大表全扫超时。

## 2. 口径参数

| 参数 | 值 | 说明 |
|---|---|---|
| 观察结束日期 | 数仓最新快照日 | `MAX(SUBSTR(cache_day, 1, 10))`，当前为 `2026-06-29` |
| 观察窗口 N | 7 天 | 近 7 天 |
| 窗口起始日期 | `2026-06-23` | 结束日期 - 6 |
| 全量角色基准日 | 数仓最新快照日 | 用 `snap_rolecache` 最新快照作为分母 |

实际查询时，日期替换为 `<昨天ds>` / `<7天前ds>` / `<昨天yyyy-MM-dd>` / `<7天前yyyy-MM-dd>` 占位符，由 `workspace.py` 自动替换。

## 3. SQL 一：综合分类统计（推荐）

统计口径：以最新角色快照为全量角色池，按“是否付费 + 最近 7 天是否登录 + 最近 7 天是否付费”划分为 6 类。

```sql
WITH params AS (
  SELECT '<昨天ds>' AS end_ds, '<7天前ds>' AS start_ds,
         '<昨天yyyy-MM-dd>' AS end_date, '<7天前yyyy-MM-dd>' AS start_date
),

role_pool AS (
  SELECT DISTINCT
    role_id,
    role_name,
    server_id,
    opgame_id,
    role_level,
    role_vip,
    total_pay_money,
    last_login
  FROM gameeco_raw.v_presto_snap_rolecache
  WHERE game_id = '312'
    AND SUBSTR(cache_day, 1, 10) = '<昨天yyyy-MM-dd>'
    AND role_type = 1                       -- 只保留正常角色
    AND SUBSTR(CAST(server_id AS VARCHAR), 5, 1) != '4'
),

recent_login AS (
  SELECT
    role_id,
    MAX(ds) AS last_login_ds,
    COUNT(DISTINCT ds) AS login_days
  FROM gamelog_odl.v_presto_log_rolelogin
  WHERE game_id = 312
    AND ds >= '<7天前ds>'
    AND ds <= '<昨天ds>'
    AND SUBSTR(CAST(server_id AS VARCHAR), 5, 1) != '4'
  GROUP BY role_id
),

recent_pay AS (
  SELECT
    role_id,
    SUM(CAST(pay_money AS DOUBLE)) AS recent_pay_money,
    COUNT(*) AS recent_pay_times
  FROM gamelog_odl.v_presto_log_payrecharge
  WHERE game_id = 312
    AND ds >= '<7天前ds>'
    AND ds <= '<昨天ds>'
    AND SUBSTR(CAST(server_id AS VARCHAR), 5, 1) != '4'
  GROUP BY role_id
),

classified AS (
  SELECT
    r.role_id,
    r.role_name,
    r.opgame_id,
    r.role_level,
    r.role_vip,
    COALESCE(r.total_pay_money, 0) AS total_pay_money,
    CASE WHEN l.role_id IS NOT NULL THEN 1 ELSE 0 END AS has_login_n_days,
    l.last_login_ds,
    l.login_days,
    COALESCE(p.recent_pay_money, 0) AS recent_pay_money,
    CASE
      WHEN l.role_id IS NULL THEN '流失'
      WHEN r.total_pay_money > 0 AND COALESCE(p.recent_pay_money, 0) = 0 THEN '沉默'
      ELSE '活跃'
    END AS life_state,
    CASE WHEN COALESCE(r.total_pay_money, 0) > 0 THEN '付费用户' ELSE '免费用户' END AS pay_type
  FROM role_pool r
  LEFT JOIN recent_login l ON r.role_id = l.role_id
  LEFT JOIN recent_pay  p ON r.role_id = p.role_id
)

SELECT
  pay_type,
  life_state,
  COUNT(DISTINCT role_id) AS user_count,
  ROUND(AVG(role_level), 1) AS avg_level,
  ROUND(AVG(role_vip), 1) AS avg_vip,
  ROUND(SUM(total_pay_money), 2) AS total_pay
FROM classified
GROUP BY pay_type, life_state
ORDER BY pay_type, life_state
LIMIT 50
```

## 5. SQL 二：仅查“流失付费用户”明细

用于运营召回：找出历史付过费、但最近 N 天完全没登录的角色。

```sql
SELECT
  r.role_id,
  r.role_name,
  r.server_id,
  r.opgame_id,
  r.role_level,
  r.role_vip,
  r.total_pay_money,
  r.last_login
FROM gameeco_raw.v_presto_snap_rolecache r
WHERE r.game_id = '312'
  AND SUBSTR(cache_day, 1, 10) = '<昨天yyyy-MM-dd>'
  AND r.role_type = 1
  AND SUBSTR(CAST(r.server_id AS VARCHAR), 5, 1) != '4'
  AND COALESCE(r.total_pay_money, 0) > 0
  AND NOT EXISTS (
    SELECT 1
    FROM gamelog_odl.v_presto_log_rolelogin l
    WHERE l.game_id = 312
      AND l.role_id = r.role_id
      AND l.ds >= '<7天前ds>'
      AND l.ds <= '<昨天ds>'
      AND SUBSTR(CAST(l.server_id AS VARCHAR), 5, 1) != '4'
  )
ORDER BY r.total_pay_money DESC
LIMIT 200
```

## 7. 规则确认与校验

### 7.1 流失定义是否正确？

**正确。**

流失只看登录表 `gamelog_odl.v_presto_log_rolelogin`，不关联充值表。只要近 N 天 `ds` 范围内没有该 `role_id` 的登录记录，即判定为流失。

### 7.2 沉默定义是否正确？

**正确。**

沉默用户需要同时满足：
1. 快照累计充值 > 0（`total_pay_money > 0`）；
2. 近 N 天有登录（`EXISTS recent_login`）；
3. 近 N 天无充值（`NOT EXISTS recent_pay`）。

这样付费用户“有登录无充值”会被正确归为沉默，而不是流失。

### 7.3 免费用户是否有“沉默”概念？

业务上通常只关注付费用户的沉默。免费用户长期登录但无充值属于正常活跃用户。如需给免费用户也定义“沉默”，可同样使用“有登录但无付费”逻辑，但一般不建议单独召回。

### 7.4 数据表选择是否正确？

| 用途 | 选择表 | 原因 |
|---|---|---|
| 全量角色池 | `gameeco_raw.v_presto_snap_rolecache` | 每日每个角色一条，含 `role_type`、`total_pay_money`，适合做分母 |
| 登录行为 | `gamelog_odl.v_presto_log_rolelogin` | T+1 稳定，每次登录一条 |
| 充值行为 | `gamelog_odl.v_presto_log_payrecharge` | T+1 稳定，每笔充值一条 |

**正确。**

### 7.5 关于 `role_id` 类型

- `snap_rolecache` 中 `role_id` 是 `bigint`；
- `gamelog_*` 登录/充值表中 `role_id` 是 `string`。

Presto 会自动做类型隐式转换，当前 `EXISTS` 子查询写法已验证可运行。

### 7.6 测试服过滤

所有 SQL 均包含：
```sql
AND SUBSTR(CAST(server_id AS VARCHAR), 5, 1) != '4'
```
已按 schema 约定排除测试服。

### 7.7 观察窗口 N 的调整

只需替换占位符：
- 近 3 天：`<3天前ds>` / `<3天前yyyy-MM-dd>`
- 近 7 天：`<7天前ds>` / `<7天前yyyy-MM-dd>`
- 近 15 天：`<15天前ds>` / `<15天前yyyy-MM-dd>`
- 近 30 天：`<30天前ds>` / `<30天前yyyy-MM-dd>`

### 7.8 性能说明

早期版本使用 `hist_pay` CTE 对 `payrecharge` 全历史聚合，再 JOIN 大表，导致明细查询 9 分钟超时。当前版本改用：
- `total_pay_money` 快照字段判断历史是否付费；
- `EXISTS` / `NOT EXISTS` 子查询判断窗口期行为；
- 避免在 WHERE 中使用子查询引用 `params` CTE，确保分区过滤能下推。

实测三条 SQL 均可在数秒内返回结果。

## 9. 可扩展：增加付费分层

如需进一步细分付费用户价值，可在 classified 中增加分层：

```sql
CASE
  WHEN total_pay_money >= 100000 THEN '超R'
  WHEN total_pay_money >= 10000 THEN '大R'
  WHEN total_pay_money >= 1000  THEN '中R'
  WHEN total_pay_money > 0      THEN '小R'
  ELSE '免费'
END AS pay_tier
```

然后按 `pay_tier + life_state` 分组，可得到“大R流失/沉默”、“中R流失/沉默”等更细粒度结论。
